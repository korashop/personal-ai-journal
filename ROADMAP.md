# Product Roadmap

## Guiding Principle

Optimize for usefulness, continuity, and depth over symmetry, slogans, or fixed output structures.

## Confirmed Product Requirements

### Memory and continuity

- Every new analysis should account for prior entries and prior discussion.
- The patterns area should be a living page that updates and improves as new entries are added.
- Memory should become more cumulative over time, not reset into isolated sessions.

### Product structure

- The app should feel like three areas:
  - Capture
  - Entries
  - Patterns
- Entries and patterns should feel distinct:
  - Entries are for one entry and its local dialogue
  - Patterns are for synthesis across time

### Capture

- Image-only submission must work.
- Multiple images should be attachable in one submission.
- Phone capture should become a first-class flow.
- Long term, mobile web should support camera roll and direct camera capture cleanly.
- HEIC handling should be reliable.
- OCR should support review and cleanup before final submission.

### Entries

- The raw entry should recede after submission.
- Source text should be collapsible rather than dominating the page.
- Analysis and follow-up engagement should take visual priority.
- Entry cards should not use the first line as the title.
- Entry cards should show date plus a short, informative summary of what the entry is about.
- Entry browsing should feel easier and less bloated than a left sidebar list.
- The user should be able to edit an entry after the fact.
- The user should be able to delete an entry.
- The user should be able to re-run analysis after editing.

### Analysis UX

- The app should avoid giant descriptive headers and oversized branding.
- Section framing should be subtle and utilitarian.
- Analysis should not feel formulaic.
- Different entries should allow different response lengths and structures.
- The assistant should not always force the same named sections every time.
- The tool should optimize for usefulness, not equal-sized blocks.
- The UI should allow 3-5 generated “go deeper” options that expand a chosen theme.
- The conversation area should not duplicate the initial analysis already shown above.

### Patterns

- Patterns should not be constrained to artificially even theme sizes.
- One dominant theme can be deep; smaller themes can remain light.
- The patterns page should feel clickable and exploratory.
- Clicking a theme should open a richer “state of affairs” view:
  - what this theme is really about
  - what has been written about it
  - what has been discussed about it
  - tensions or contradictions
  - open questions
- Patterns should eventually incorporate Readwise and Snipd only when they strengthen an active thread.
- Patterns should avoid empty filler boxes and invented-sounding labels.

### Visual design

- Remove the oversized “A thinking partner…” hero.
- Reduce explanatory slogans across the UI.
- Improve the color palette and overall aesthetic.
- Make the interface feel calmer, tighter, and more intuitive to move through.

## Build Sequence

### Phase A

- Compact top bar
- Better entry summaries and browsing
- Less duplicative thread UI
- Flexible analysis rendering
- Clickable deep-dive options
- More grounded patterns overview

### Phase B

- Entry edit / delete / re-analyze
- Theme detail drill-downs
- Theme-to-entry linking
- Better memory refresh behavior in the UI

### Phase C

- Real OCR review flow
- Better mobile capture
- HEIC handling improvements
- Deployed mobile-first use
- Public URL access so the app can be used from phone/browser outside localhost
- Smooth upload flow on mobile Safari/Chrome

### Phase D

- Readwise integration
- Snipd integration
- Cross-theme exploration and richer historical synthesis
